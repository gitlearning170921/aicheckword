# 版本兼容问题排查与加固（可复用流程）

## 目标

把“环境差异导致的故障”转化为“有检测、有回退、有提示”的稳定实现，减少重复返工。

## 1) 快速归因（先分三类）

- **API 变更/参数不兼容**：某函数/构造器参数在当前版本不存在（典型：框架升级/降级）。
- **网络资源不可达**：CDN、外部脚本、favicon、模型服务地址不可达导致页面空白或超时。
- **运行时行为差异**：编码/路径/权限/代理/SSL、Windows/PowerShell 引号差异等。

## 2) 收集“最低成本证据”

尽量用最小代价拿到事实：

- 打印/记录关键库版本：`fastapi.__version__`、`pydantic.__version__` 等
- 对关键 capability 做 `hasattr` 检测（例如某个参数或属性是否存在）
- 对关键 URL 做一次可达性探测（若是前端页面，优先在 HTML 中加入加载失败提示）

## 3) 加固实现的标准做法

### 3.1 capability check（推荐）

- 在调用新 API 前先判断能力是否存在；不存在则走旧实现。

### 3.2 try/except fallback（推荐）

- 先尝试新路径，失败捕获“已知可预期异常”，转老路径；
- 捕获后要返回**可读错误**（说明“是版本/网络/参数问题”）。

### 3.3 适配层隔离（强烈建议）

- 把兼容逻辑集中在 `*_compat.py` 或单独函数里，上层只调用统一接口，避免兼容逻辑散落全项目。

## 4) Swagger/Docs 白屏专用清单

- 不依赖单一 CDN；至少提供 2-3 个备选域名。
- JS/CSS 加载失败时必须在页面显示错误信息（不要白屏）。
- 如客户环境长期拦截外网：切换到“自托管静态资源”。

## 5) 完成标准（必须同时满足）

- 不同环境下至少能走到“成功”或“明确失败提示”（不允许空白/静默失败）。
- 改动不破坏内部接口（仅对外文档/可见性变化要可控）。
- 改完立刻做最小自测：语法检查 + 关键路由/关键函数调用。
