# 追溯矩阵（RTM）自动编写与自动修订（严格不发散）

本 Skill 用于：生成或修复《软件可追溯性分析报告》中的追溯矩阵（或等价表格），让输出满足本系统“跨文档可追溯性”审核的硬约束：**编号原样复制、证据可对照、闭环不虚构、范围与 SRS 一致**。

## 0) 必须先确认的输入（缺一不可就只能输出 TBD）

- **需求来源**：SRS/软件需求规范（至少包含 URS/REQ/SR 的编号与描述，表格也算）。
- **风险来源**：风险分析/风险管理（至少包含 CS/风险需求/控制措施编号列；Excel 常在“风险需求/Risk demand”列）。
- **追溯载体**：RTM/追溯矩阵表（列结构、行主键、编号规则、是否允许一对多）。
- **项目边界**：适用范围/系统功能摘要/项目名称产品型号（用于检查追溯报告文字是否越界）。

若缺少其中任意一项：禁止“自动补全闭环”，改为输出 `tbd` 列表与可执行补充建议。

## 1) 硬约束（必须遵守）

### 1.1 编号逐字复制（最高优先级）

- 任何 ID 必须逐字复制原文：`URS001` ≠ `URS1`；`CE01` ≠ `CS01`；`CS09` 不能写成 `CS9`。
- 若输入出现 `【以修订后为准】CSxx（已废止 Cyyy/…）`：仅以 `CSxx` 为当前生效编号；不得拼接成 `CS012` 等虚构编号。

### 1.2 闭环不虚构

- 只能建立“你能在输入中找到证据”的链接。
- 不能为了“看起来完整”而补写不存在的设计项/代码模块/测试用例编号。
- 不能把“相似表述”当作“同一需求/同一风险”强行对齐；缺证据就标注待确认。

### 1.3 不发散（范围与 SRS 绑定）

- RTM 的文字段落（非表格）对系统功能、适用范围、需求来源、风险控制的叙述必须与 SRS/适用范围一致。
- 不新增适应症/功能模块/用户群/外部接口等项目未声明内容。

### 1.4 定位可审计（避免页码）

- 不写虚构页码；用章节/表格标题/表头/行编号定位。

## 2) 追溯矩阵的推荐列模型（可映射到你现有模板）

优先保证以下信息能被核对（列名可不同，但语义要对应）：

- **需求列**：`URS/REQ/SR ID` + `需求描述摘要`
- **风险列**：`CS/风险需求(控制措施) ID` + `风险/控制措施描述摘要`
- **设计列**：`设计项/架构组件/模块 ID`（若组织有编码体系）
- **实现列（可选）**：代码仓/模块路径/配置项（若项目有受控引用）
- **测试列**：`测试用例 ID` / `验证记录 ID` + `验收点摘要`
- **证据/来源列（强烈建议）**：对应来源文档名 + 表格列名/章节名（不含页码）
- **状态列（修订场景）**：新增/修改/删除 + 修订原因

## 3) 生成流程（从证据到矩阵）

### Step A：先做“编号登记册”（禁止漏检）

- 从 SRS 表格与正文中提取所有 URS/REQ/SR 编号（逐字）。
- 从风险分析中提取所有 CS 编号：优先从 Excel 的“风险需求 / Risk demand (Measure) ID”列按单元格原文提取。
- 如存在追溯报告既有矩阵：提取其行主键与各列编号，作为“待修复清单”。

输出一个 `id_registry`（只列编号，不解释）用于后续对齐与防漏。

### Step B：建立“最小可证据闭环”

对每个 URS/REQ 行：

- 找到其引用的 CS（若 SRS 追溯表已有 CS 列，则以其为准）。
- 对每个 CS：
  - 在风险分析中验证该 CS 存在（按原文匹配）。
  - 对比风险控制措施文字与需求意图是否明显矛盾（若矛盾，标注需确认/需修订）。
- 如存在设计/测试编号：
  - 仅在输入出现对应编号时填入；否则留空并在 `tbd` 里提出需要补充哪类证据（例如“需提供系统测试用例编号清单”）。

### Step C：一致性与越界检查（写之前先查）

- 追溯报告文字描述是否与 SRS 的适用范围/功能一致。
- 若存在项目基本信息（项目名/型号/产品名）：追溯报告封面/标题/引用中需一致。

## 4) 修订/修复流程（用于“自动修改”）

当输入包含“现有 RTM + 新的 SRS/风险分析版本”时：

- **先对齐编号变更**：若出现修订标记（如 `【以修订后为准】`），把矩阵中引用的旧编号替换为新编号，并在状态列记录“替换原因：修订生效”。
- **再做差异修复**：
  - SRS 新增/删除需求 → 增删行
  - 风险分析新增/废止 CS → 更新风险列，并标注需要同步的 SRS/测试条目
  - 发现“矩阵有号无文” → 输出为缺失项（不是编造补齐）

## 5) 输出格式（便于程序写回 Word/Excel）

请输出一个 JSON，对应“矩阵行 + 修复动作 + 待补充项”：

```json
{
  "id_registry": {
    "requirements": ["URS001", "REQ-012"],
    "risks": ["CS09", "CS11"],
    "tests": ["TC-001"]
  },
  "rows": [
    {
      "requirement_id": "URS001",
      "requirement_summary": "…",
      "risk_id": "CS09",
      "risk_summary": "…",
      "design_ref": "",
      "test_ref": "TC-001",
      "evidence": [
        {"doc": "SRS", "where": "【Word 表格结构化摘录】URS列=URS001"},
        {"doc": "风险分析", "where": "【Excel 工作表：风险分析】列=风险需求 单元格原文=CS09"}
      ],
      "status": "新增|修改|删除|不变",
      "notes": ""
    }
  ],
  "issues": [
    {
      "type": "missing-link|id-mismatch|scope-conflict",
      "severity": "high|medium|low|info",
      "description": "…",
      "suggestion": "…"
    }
  ],
  "tbd": ["仍缺少的证据/资料清单…"]
}
```

## 6) 风格要求（让审核员看起来真实）

- 用“证据驱动”的克制语气：能核对的就写明来源；不能核对的就列 TBD，不自嗨式闭环。
- 文字描述避免模板口号，用“范围+机制+证据”的写法。
