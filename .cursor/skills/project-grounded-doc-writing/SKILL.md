# 基于项目实际与知识库的文档编写（不发散）

## 目标

输出“审核员看起来真实、可对照”的文档内容：**只写有依据的事实**，以项目范围与知识库要求为边界，避免扩写发散。

## 输入（优先级从高到低）

1. **项目约束**：注册国家、注册类别、项目形态、文档语言、适用范围、系统功能识别摘要、项目基本信息（项目名/产品名/型号等）。
2. **知识库/程序/法规**：作为“必须覆盖的要素/禁止项/格式规则/证据要求”，不是让你扩写成百科。
3. **待编写文档的上下文**：已有章节、术语表、表格字段、编号体系、版本与修订历史。

## 生成流程（必须按步骤执行）

### Step A：先建立“不可越界清单”

- **范围边界**：适用范围里没出现的功能/用途/用户/场景 → 不写。
- **证据边界**：无项目资料或知识库依据的结论 → 不下断言，改为“需补充/待确认”。
- **编号边界**：编号体系不清 → 不生成新编号；只引用已存在编号并逐字复制。

### Step B：选择“写作单位”

只写用户需要的最小单位之一：

- 章节（含小节）
- 表格的一行/一列定义
- 追溯矩阵的一行说明（仅在能闭环时）
- 修订说明（明确改动点与原因）

### Step C：输出模板（按文档语言）

#### 中文段落模板（工程文档风格）

- 先给**一句结论/定义**（可被对照）
- 再给**2–5 条要点**（每条都能在项目资料/系统实际/知识库中对应）
- 如需引用要求：用“依据/参照”而非泛化口号

#### 英文段落模板（避免中文审核腔）

- One-sentence definition
- Bullet points with verifiable specifics (scope, interfaces, constraints, acceptance criteria)
- “TBD/To be provided” for missing evidence (not hallucination)

## 与当前系统审核硬约束的对齐点（写作时强制遵守）

- **页码**：不要写“第X页”，除非明确存在页码标记。
- **编号逐字**：URS001/CE01/CS09 等必须逐字复制（含前导零与前缀）。
- **表格同等效力**：Word/Excel 表格字段是审核重点；表头/列名要清晰且一致。
- **时间线一致**：版本号/生效日期/修订日期不可自相矛盾。

## 输出要求（给下游“自动修改/自动填充”用）

输出时按以下结构组织，便于程序落库或自动写入：

- **content**：最终要写入文档的文本（不含解释）。
- **assumptions**：你做了哪些假设（每条一句）。
- **evidence_used**：用到的依据（项目资料摘录/知识库要点/已有章节）。
- **open_questions**：仍缺的信息（可直接变成待办或表格空格）。

示例格式：

```json
{
  "content": "……",
  "assumptions": ["……"],
  "evidence_used": ["……"],
  "open_questions": ["……"]
}
```
