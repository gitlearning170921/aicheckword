# 审核系统扩展设计方案：项目专属 + 上下文识别

> **已实现**（按需求 1～4）：注册国家/项目形态可配置，注册类别/注册组成固定选项，项目多选维度，多项目列表，仅通用/按项目审核，选项目后可临时修改维度。

## 一、目标简述

审核时需同时满足：

1. **通用要求**：法规、程序文件、项目案例（已有）
2. **项目专属要求**：该项目的**技术要求、说明书指定要求**等
3. **按上下文筛选**：
   - 按**注册国家、注册类别**识别适用的法规和程序要求
   - 按**项目形态**（Web / APP / PC）识别适用的项目案例要求

---

## 二、概念与数据模型

### 2.1 现有概念（保持不变）

| 概念 | 说明 | 存储 |
|------|------|------|
| 法规知识库 | 法规 / 程序 / 项目案例 | Chroma + MySQL `knowledge_docs`（含 category） |
| 审核点清单 | 通用审核点（由法规知识库生成） | Chroma 审核点集合 + MySQL `audit_checklists` |
| 审核点知识库 | 已训练的审核点向量 | `{collection}_checkpoints` |

### 2.2 新增概念

| 概念 | 说明 | 建议存储 |
|------|------|----------|
| **项目 (Project)** | 一个具体申报/产品项目，带注册维度 | MySQL 新表 `projects` |
| **项目专属资料** | 该项目独有的技术要求、说明书等文件 | 新向量库（或同库+metadata） |
| **审核上下文** | 单次审核时的 国家/类别/形态 | 审核时选择或从项目带出 |

### 2.3 项目 (Project) 字段建议

| 字段 | 类型 | 说明 | 是否必填 |
|------|------|------|----------|
| id | 主键 | - | - |
| name | 名称 | 项目名称 | 必填 |
| registration_country | 注册国家 | 如：中国、美国、欧盟 | 必填（审核时用于筛法规/程序） |
| registration_type | 注册类别 | 如：二类医疗器械、三类、软件 | 必填（同上） |
| project_form | 项目形态 | web / app / pc（可扩展） | 必填（用于筛项目案例） |
| collection | 所属知识库 | 与现有 collection 一致 | 必填 |
| created_at / updated_at | 时间 | - | - |

### 2.4 项目专属资料存储方式（两种可选）

**方案 A：独立向量集合（推荐）**

- 每个项目一个 Chroma 集合：`{collection}_project_{project_id}`
- 文档块带 metadata：`project_id`, `doc_type`（技术要求/说明书/其他）
- 审核时：检索「通用审核点」+「当前项目的项目专属集合」

**方案 B：统一集合 + metadata 过滤**

- 一个集合如 `{collection}_project_docs`，每条 chunk 带 metadata：`project_id`, `registration_country`, `registration_type`, `project_form`, `doc_type`
- 审核时按 metadata 过滤（需向量库支持 where 条件）

建议先用**方案 A**，实现简单、隔离清晰。

## 三、流程改造

### 3.1 第一步：法规训练 & 生成审核点（基本不变）

- 继续支持：上传 法规/程序/项目案例，选择分类，训练到法规知识库。
- **可选增强**（若您需要按国家/形态筛选法规）：
  - 训练时为每条文档打标签：适用国家、适用注册类别、适用项目形态（如“通用”）。
  - 需您确认：是否在第一步就做“适用国家/类别/形态”的标注（人工选择 or 后续再做）。

### 3.2 第二步：审核点训练（扩展）

- **2a. 通用审核点清单**（现有）
  - 生成/导入审核点清单 → 训练到「审核点知识库」。
- **2b. 项目专属资料**（新增）
  - 选择或新建**项目**（名称、注册国家、注册类别、项目形态）。
  - 上传该项目专属文件：**技术要求、说明书**等。
  - 支持格式：与现有一致（PDF/Word/Excel/TXT/Markdown 等）。
  - 可选：为每个文件选择**资料类型**（技术要求 / 说明书 / 其他），写入 metadata 便于后续展示或筛选。

### 3.3 第三步：文档审核（扩展）

- **选择审核模式**：
  - **仅通用审核**：不选项目，仅用「审核点知识库」。
  - **按项目审核**：选择**项目**，系统自动带出 注册国家、注册类别、项目形态。
- **检索策略**：
  - 始终检索：**审核点知识库**（通用审核点）。
  - 若选了项目：再检索**该项目专属向量库**，将「通用审核点 + 项目专属要求」一起拼进 prompt。
- **Prompt 增强**：
  - 在 prompt 中显式说明：本次审核的**注册国家、注册类别、项目形态**，要求模型同时依据通用审核点与项目专属技术要求/说明书进行审核。

### 3.4 法规/程序/项目案例的“按上下文筛选”（可选）

若希望在检索**法规知识库**时也按国家、类别、形态缩小范围，需要：

- 在**第一步训练**时，为 法规/程序/项目案例 打上适用维度（例如适用国家、适用注册类别、适用项目形态）。
- 审核时传入 国家/类别/形态，检索法规知识库时带 metadata 过滤（或仅在「生成审核点」时按维度筛选）。

这部分可以放在**二期**：先实现「通用审核点 + 项目专属资料」，再考虑法规库的维度标签与过滤。
